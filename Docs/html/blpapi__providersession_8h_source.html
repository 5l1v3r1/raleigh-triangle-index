24 //@PURPOSE: Provide a session that can be used for providing services
25 //
26 //@CLASSES:
27 // blpapi::ProviderSession: Session with providing(publishing) services.
28 // blpapi::ProviderEventHandler: Event handler for ProviderSession
29 // blpapi::ServiceRegistrationOptions: Container holding registration options.
30 //
31 //@DESCRIPTION: ProviderSession inherits from AbstractSession. In addition to
32 // AbstractSession functionality, ProviderSession provides functions that are
33 // needed to support publishing like 'registerService', 'createTopics' and
34 // 'publish'.
35 //
36 // Topic Life Cycle
37 // ----------------
38 // A provider wishing to publish subscription data must explicitly open each
39 // topic on which they publish using 'ProviderSession::createTopics' (or
40 // 'ProviderSession::createTopicsAsync'). Creating a topic prepares the
41 // distribution and caching infrastructure for new data associated with the
42 // topic's resolved identifier. (Note that several different topics could
43 // resolve to the same ID.) Independent of a topic's creation status is its
44 // subscription status, i.e. whether there are subscribers ready to receive
45 // the data published. A topic that is both created and subscribed is
46 // *activated*.
47 //
48 // There are two models for managing topic creation: broadcast and
49 // interactive. Broadcast publishers proactively call
50 // 'ProviderSession::createTopic*' for each topic on which they intend to
51 // publish, while interactive publishers wait to receive a
52 // 'TopicSubscribed' message (within an 'Event' of type 'Event::TOPIC_STATUS')
53 // before calling 'ProviderSession::createTopic*' in response. Topics are
54 // resolved before they are created---it is possible that multiple different
55 // topic strings will map to the same underlying topic. See below for the
56 // behavior of the SDK when the same topic is created multiple times.
57 //
58 // After 'ProviderSession::createTopic*' is called, the publisher will receive
59 // a 'TopicCreated' message (within an 'Event::TOPIC_STATUS' event), and when
60 // there is at least one subscriber to the topic the publisher will then
61 // receive a 'TopicActivated' message (also within an 'Event::TOPIC_STATUS'
62 // event). As subscribers come and go, additional 'TopicSubscribed',
63 // 'TopicActivated', 'TopicUnsubscribed', and 'TopicDeactivated' messages may
64 // be received by the publisher. A 'Topic' object can be retrieved from each of
65 // these messages using the 'ProviderSession::getTopic' method, and this
66 // object can be used for subsequent calls to 'EventFormatter::appendMessage'
67 // and 'ProviderSession::deleteTopic'. In the case that the same resolved
68 // topic is created multiple times by a publisher using different names, it is
69 // unspecified which of those names will be returned by 'Message::topicName'
70 // for these (or other) messages.
71 //
72 // If a publisher no longer intends to publish data on a topic, it can call
73 // 'ProviderSession::deleteTopic*' to free the internal caching and
74 // distribution resources associated with the topic. When a resolved topic has
75 // been deleted the same number of times that it has been created,
76 // a 'TopicDeleted' message will be delivered, preceded by 'TopicUnsubscribed'
77 // and 'TopicDeactivated' messages if the topic was still subscribed (and
78 // activated). No further messages can be published on a deleted topic.
79 //
80 // Deregistering a service deletes all topics associated with that service.
81 //
82 // Note that 'TopicActivated' and 'TopicDeactivated' messages are entirely
83 // redundant with 'TopicCreated', 'TopicSubscribed', 'TopicDeleted', and
84 // 'TopicUnsubscribed' messages, and are provided only for the convenience
85 // of publishers that do not maintain per-topic state.

504  // This class provides a session that can be used for providing services.
505  //
506  // It inherits from AbstractSession. In addition to the AbstractSession
507  // functionality a ProviderSession provides the following
508  // functions to applications.
509  //
510  // A provider can register to provide services using
511  // 'ProviderSession::registerService*'. Before registering to provide a
512  // service the provider must have established its identity. Then
513  // the provider can create topics and publish events on topics. It also
514  // can get requests from the event queue and send back responses.

516  // After users have registered a service they will start receiving

523  // priority for the sub-service code will receive subscription requests; if
524  // multiple providers have registered the same sub-service code with the
525  // same priority (or the resolver did not set a sub-service code for the
526  // subscription), the subscription request will be routed to one of the
