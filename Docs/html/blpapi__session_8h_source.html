295 class Session : public AbstractSession {
296  // This class provides a consumer session for making requests for Bloomberg
297  // services.
298  //
299  // Sessions manage access to services either by requests and
300  // responses or subscriptions. A Session can dispatch events and
301  // replies in either a synchronous or asynchronous mode. The mode
302  // of a Session is determined when it is constructed and cannot be
303  // changed subsequently.
304  //
305  // A Session is asynchronous if an EventHandler object is
306  // supplied when it is constructed. The setEventHandler() method
307  // may be called to adjust the way events are handled subsequently
308  // and the nextEvent() method may not be called. All incoming
309  // events are delivered to the EventHandler(s) supplied on
310  // construction or subsequently using setEventHandler().
311  //
312  // A Session is synchronous if an EventHandler object is not
313  // supplied when it is constructed. The nextEvent() method must be
314  // called to read incoming events and the setEventHandler() method
315  // may not be called.
316  //
317  // Several methods in Session take a CorrelationId parameter. The
318  // application may choose to supply its own CorrelationId values
319  // or allow the Session to create values. If the application
320  // supplies its own CorrelationId values it must manage their
321  // lifetime such that the same value is not reused for more than
322  // one operation at a time. The lifetime of a CorrelationId begins
323  // when it is supplied in a method invoked on a Session and ends
324  // either when it is explicitly cancelled using cancel() or
325  // unsubscribe(), when a RESPONSE Event (not a PARTIAL_RESPONSE)
326  // containing it is received or when a SUBSCRIPTION_STATUS Event
327  // which indicates that the subscription it refers to has been
328  // terminated is received.
329  //
330  // When using an asynchronous Session the application must be
331  // aware that because the callbacks are generated from another
332  // thread they may be processed before the call which generates
333  // them has returned. For example, the SESSION_STATUS Event
334  // generated by a startAsync() may be processed before
335  // startAsync() has returned (even though startAsync() itself will
336  // not block).
337  //
338  // This becomes more significant when Session generated
339  // CorrelationIds are in use. For example, if a call to
340  // subscribe() which returns a Session generated CorrelationId has
341  // not completed before the first Events which contain that
342  // CorrelationId arrive the application may not be able to
343  // interpret those events correctly. For this reason, it is
344  // preferable to use user generated CorrelationIds when using
345  // asynchronous Sessions. This issue does not arise when using a
346  // synchronous Session as long as the calls to subscribe() etc are
347  // made on the same thread as the calls to nextEvent().
348

379  // Construct a Session using the optionally specified
380  // 'options', the optionally specified 'eventHandler' and the
381  // optionally specified 'eventDispatcher'.
382  //
383  // See the SessionOptions documentation for details on what
384  // can be specified in the 'options'.
385  //
386  // If 'eventHandler' is not 0 then this Session will operation
387  // in asynchronous mode, otherwise the Session will operate in
388  // synchronous mode.
389  //
390  // If 'eventDispatcher' is 0 then the Session will create a
391  // default EventDispatcher for this Session which will use a
392  // single thread for dispatching events. For more control over
393  // event dispatching a specific instance of EventDispatcher
394  // can be supplied. This can be used to share a single
395  // EventDispatcher amongst multiple Session objects.
396  //
397  // If an 'eventDispatcher' is supplied which uses more than
398  // one thread the Session will ensure that events which should
399  // be ordered are passed to callbacks in a correct order. For
400  // example, partial response to a request or updates to a
401  // single subscription.
402  //
403  // If 'eventHandler' is 0 and the 'eventDispatcher' is not
404  // 0 an exception is thrown.
405  //
406  // Each EventDispatcher uses its own thread or pool of
407  // threads so if you want to ensure that a session which
408  // receives very large messages and takes a long time to
409  // process them does not delay a session that receives small
410  // messages and processes each one very quickly then give each
411  // one a separate EventDispatcher.

682  //

687  // being used. The SubscriptionIterator is guaranteed to never
688  // return the same subscription twice. However, the subscription
689  // the iterator points to may no longer be active. In this case
690  // the result of subscriptionStatus() will be UNSUBSCRIBED or
691  // CANCELLED.
